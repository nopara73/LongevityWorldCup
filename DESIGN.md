# LongevityWorldCup Design Decisions

This file records durable UI decisions for Longevity World Cup. Keep it short: add only reusable product-design guidance, not implementation history, workflow rules, or one-off polish notes.

## Visual identity: Scoreboard

The visual direction is dark scoreboard chrome around light, breathing content. The dark look belongs to the chrome (header band, hero, sticky bar, footer, athlete modal); page content — including the homepage below the hero — always sits on the light paper surface. Never put content sections on a black page background. The shared tokens live in the `:root` block of `wwwroot/partials/header.html`; standalone pages (error pages, privacy policy, embeds) carry local copies that must stay in tune with them.

- Dark chrome (header, footer, hero, modal): near-black `#050908`/`#030706` surfaces, off-white ink `#f8f7ef`, bright green `#43ee83` reserved for scores, numerals, and the primary CTA. Dark surfaces are never flat black: the header band carries a faint green top sheen and the athlete modal a diagonal charcoal gradient (`linear-gradient(120deg, #070b09, #2c3530 130%)`) so they read as lit surfaces. Hairline borders `rgba(248, 247, 239, 0.14)` separate chrome from content instead of drop shadows.
- Light pages invert that: warm paper `#f4f3ea` background, near-black warm ink `#1c2722` text, dark green `#0b7d45` (`--primary-color`) for links and accents, with green tints `rgba(11, 125, 69, x)` for borders, hovers, and panels.
- The secondary accent is amber/gold (`--secondary-color: #8f6a0e`, tints `rgba(212, 175, 55, x)`), used for crowd/Guess-My-Age features, Amateur tier marking, prize semantics, and new-entry cues. Do not reintroduce cyan, pink, or Material blue/green as accents.
- Color roles are semantic and restrained: green is the signature action/current-state color; amber is competition secondary status (Amateur, crowd, prize, new athlete); red is danger/destructive; neutral sage/ink handles inactive chrome and body surfaces. Blue/purple/pink are not brand colors and may appear only inside literal third-party platform previews or logos where the platform identity is the content.
- Display type is the condensed `--display-font` stack, uppercase, used for page `h1` titles, the wordmark, section headings, and big numerals. The stack ships `Anton` (self-hosted woff2, loaded via the `{{ASSET_FONT_ANTON}}` middleware placeholder) so the voice survives on iOS/Android, with Impact/Haettenschweiler/Arial Narrow Bold as fallbacks. Anton has a single weight: declare `font-weight: 400` with `font-synthesis: none` wherever the display font is used — never bold it. Body and UI text stay Roboto.
- Typography uses the shared tokens in `wwwroot/partials/header.html`: `--text-caption`, `--text-meta`, `--text-small`, `--text-ui`, `--text-body`, `--text-lead`, `--text-title`, plus `--leading-tight`, `--leading-ui`, `--leading-body`, and `--leading-reading`. New UI should consume these tokens instead of inventing local `0.9rem`/`1.2rem` sizes and arbitrary line-heights. Body copy defaults to `--leading-body`; long docs and rules copy uses `--leading-reading` with a roughly 68ch measure.
- Warm light-surface text roles are tokenized: `--text-soft` for normal explanatory copy that should sit below headings, `--text-muted` for secondary UI labels/links, and `--text-subtle` for tertiary labels, dividers, and helper metadata. Do not add local gray/slate values such as `#555`, `#334155`, `#64748b`, or one-off warm grays when one of these roles fits.
- Display type should create official scoreboard moments, not decorate every label. Use it for the wordmark, primary page titles, section headers, score numerals, rank callouts, and poster-scale statements. Do not use Anton for paragraphs, form labels, table body cells, helper text, docs prose, or dense modal metadata.
- The single brand CTA fill is the bright green gradient pill (`--cta-green-gradient`, near-black text): the header/sticky PLAY pill, homepage hero Apply action, and `.option-button.green` forward actions (Next, Continue, Start professional, I'm a new athlete). Other primary actions are solid ink pills (`--main-action-color`); secondary actions are solid muted sage (`.option-button.grey`, `#5d6f66`) or quiet outline pills (back buttons). When a secondary option sits directly beside a primary choice, use an active outline/surface treatment instead of a gray filled pill so it does not read as disabled. Hover/focus feedback uses `--cta-green-shadow-hover` plus a 1px lift, never scale/rotate effects or glow animations.
- Secondary action fills use `--secondary-action-bg`, `--secondary-action-bg-hover`, and `--secondary-action-shadow`; do not restyle the same "manage/back/secondary" action with local sage values or independent shadows.
- Reusable light-surface primitives use the shared tokens in `wwwroot/partials/header.html`: `--radius-card` (8px), `--radius-panel` (10px), `--radius-control` (pill controls), `--surface-border`, `--surface-border-strong`, `--surface-shadow`, `--surface-shadow-lifted`, `--surface-row-alt`, `--surface-row-hover`, `--surface-green-tint`, `--surface-green-tint-strong`, `--surface-amber-tint`, `--surface-amber-tint-strong`, `--focus-ring`, and disabled-control tokens. New shared CSS should consume these tokens instead of inventing local corner radii, shadows, focus rings, or disabled colors.
- Text fields, selects, and textareas use `--radius-field` for their modest input radius. Cards and panels use `--radius-card`/`--radius-panel`; pills use `--radius-control`. Do not let form controls drift into card radii or custom rounded rectangles.
- Dark chrome primitives use the shared tokens `--chrome-border`, `--chrome-border-strong`, `--chrome-panel-bg`, `--chrome-panel-bg-strong`, `--chrome-overlay-bg`, `--chrome-overlay-bg-soft`, `--chrome-shadow`, and `--chrome-shadow-soft`. Top-level page/dialog overlays use the chrome overlay slots with restrained 2px blur; do not invent local black overlays or heavy backdrop blur.
- The CTA green system is tokenized as `--cta-green-gradient`, `--cta-green-shadow`, and `--cta-green-shadow-hover`. Do not recreate the same gradient/shadow values inside local components; reference the token with a fallback only on standalone pages that may render without the shared header partial.
- Reusable form and chooser components use the shared control tokens in `wwwroot/partials/header.html`: `--control-bg`, `--control-border`, `--control-border-hover`, `--control-border-focus`, `--control-hover-bg`, `--control-focus-ring`, plus `--error-*` status tokens. Autocomplete panels, selects, text inputs, crop/preview frames, and upload confirmation panels should use these tokens before hard-coded white/slate/red values.
- Content cards, dense data shells, event boards, challenge panels, radar panels, and modal sub-panels should feel like official record surfaces: hairline border, 8-10px radius, low shadow, warm paper fill. Avoid the ghost-card pattern of a 1px border plus a wide `0 10px 30px` shadow, and avoid 15-24px card radii unless the element is a special badge/token shape or the dark athlete modal chrome.
- Brand-specific surfaces should not look like SaaS cards, wellness-blog panels, crypto dashboards, or startup newsletter blocks. When a generic module is needed, frame it in competition language: records, standings, rules, prize pool, season bulletin, proof, athletes, champion payouts. Use scoreboard display type, tight official-record borders, and restrained sport-surface hierarchy instead of soft white cards, broad shadows, scale-on-hover buttons, and vague subscription/support copy.
- Official competition attributes use compact record badges or table fields, not sales-style decoration. Avoid diagonal corner ribbons, tilted stickers, coupon banners, and oversized track labels; prize, track, and proof cues should support the record without covering athlete imagery or competing with rank/score.
- Legacy accent names and values are retired. Do not introduce `cyan`, `pink`, Material blue, or blue-gray utility accents for ordinary UI. Use `--primary-color` for active/selected/linked competition affordances, `--secondary-color` for amateur/crowd/prize semantics, and explicit status tokens or local `--danger` only for errors. Standalone pages that cannot load shared partials should still copy the same token names and surface rules.
- Shared status colors live with the global palette tokens (`--success-*`, `--danger-*`, `--warning-*`) and should be reused before adding hard-coded reds/greens/yellows. Proof success panels, upload confirmations, validation errors, and destructive actions must use these tokens so contrast and tone stay consistent.
- Pheno/Bortz/clock badge colors stay in the green/ink/amber brand family. Do not use Material blue for Pheno badges or ordinary proof/category chips; preserve podium metals and genuine status/category colors only where the color carries competition meaning.
- Do not use thick colored side stripes to communicate state on cards, panels, or list items. Use a full hairline border, surface tint, badge, icon, or inset full-perimeter ring instead. Color should identify the state, not decorate one edge.
- Primary competition language should feel like a serious global sport: `standings`, `compete`, `athlete record`, `official rules`, `public proof`, `champions`, `prize pool`, and `season record`. Avoid casual game/tracking phrases such as `just track it`, `join game`, or playful motivational copy when an official competition action is clearer.
- Status colors (success green, danger red, warning amber) and categorical colors (habit icons, badge tiers, podium gold) stay semantic and are not part of the brand accent system.

## Principles

- Text should earn its place. Do not add explanatory UI copy, labels, hints, subtitles, or helper text unless the user would otherwise hesitate, misunderstand, or fail. Prefer clearer layout, stronger affordance, better grouping, or more obvious controls before adding words.

- Static fallback copy must be production copy. Pre-JS dialogs, hidden success states, empty states, and server-rendered fallbacks must not contain fake addresses, sample names, lorem/demo values, or jokey placeholders; if dynamic data is unavailable, use neutral official wording that still feels safe when exposed.
- Public form placeholders should name the field or accepted contact type, not show fake sample addresses or demo URLs. Use `Email address`, `Contact email`, `Public site or profile URL`, or `Media email or public contact link` instead of `you@example.com`, `athlete@example.com`, or similar template values.

- Major pages and dialogs must present hierarchy in this order: primary message, primary action, proof/context, secondary actions, then details. If a surface is about competition status, rank, score, athlete identity, and proof must appear before badges, biographies, decorative media, or long supporting tables. If a surface is a task flow, the title, required inputs, and continuation controls outrank brand chrome and illustration.

- Spacing uses the shared 4pt-derived scale in `wwwroot/partials/header.html` (`--space-2xs` through `--space-3xl`). Related controls and labels stay tight (`--space-xs`/`--space-sm`), component interiors use `--space-md`/`--space-lg`, and distinct sections separate with `--space-2xl` or more. Avoid one-off margins unless a component has a clear optical need.

## Decisions

- The homepage hero is a full-bleed competition-first poster on phones (≤640px wide), but desktop and tablet must never be poster-only. At 641-1023px wide the first viewport uses a stacked scoreboard: record score, compact live standings with athlete faces/scores, stats, and CTAs. At ≥1024px wide the first viewport is a split scoreboard: the record score, proof stats, and Join CTA on the left; a live standings arena on the right that mirrors the generated top-three podium with real athlete faces, ranks, scores, prizes, and a leaderboard link. The score may stay huge only when it supports that competition context.
- Desktop first viewport composition must read as a live sport, not a poster: the hero headline names the World Cup record, the primary Join action and live-leaderboard action sit together below the stats, and the live standings arena scales up on wide desktops so athlete faces, ranks, scores, proof/prize cues, and the full-leaderboard path remain first-screen content.
- The desktop homepage still needs an unmistakable Longevity World Cup brand signal above the scoreboard. The compact dark brand bar may stay fixed and space-efficient, but its phoenix and wordmark must be legible at a glance on laptop screenshots; do not shrink it into incidental chrome.
- Mobile first viewport must not be a cropped desktop poster. It should show the World Cup record, a compact live top-three standings strip with athlete faces/ranks/scores, current competition stats, Join and live-leaderboard actions, and at least a visible handoff into the light page content on common phone heights.

- The conversion pitch ("Too old for your sport? Not this one…") is the first line after the hero poster, sitting on the light paper above the podium with clear air below the dark block. Do not hide or fold it into the hero: the hero sells the score, the pitch line sells the game. It is set one step larger than body copy so it reads as a statement, not a caption.

- Homepage body sequence should move from product reality to proof to support, not from product to decoration. Keep the order as: hero, pitch, podium/leaderboard preview, live activity plus Hall of Fame, FAQ/rules path, prize-pool/supporter surfaces, newsletter, footer. Store/supporter content belongs near contribution and newsletter, not directly after the leaderboard before the competition record is established.
- Paired homepage support surfaces should align as real content modules, not equal-height placeholders. If one side is shorter, let it keep its natural height; do not stretch a card into a large blank lower half just to make a row rectangle.
- Supporter-kit merchandise is a secondary competition-support module, not a storefront hero. Desktop may show real product cards, but phone layouts should expose all items as compact support rows instead of hiding them behind an auto-rotating carousel.

- The hero subtitle states what the product is ("The world championship of age reversal."), never a generic label. Label-like copy ("longevity leaderboards") belongs to the wordmark sublines in the chrome, not to the hero.

- Podium cards lead with the score, not the prize: the years-reduced numeral is a display-font scoreboard moment (block numeral over a small uppercase caption) that visually outranks the prize money panel below it. Card heights are fixed per rank on desktop for the stair effect — when the score block grows, grow the heights with it so nothing clips into the prize panel (on phones, grow `--podium-prize-panel-height` instead).

- Money never appears unlabeled: the podium prize panel carries a small uppercase "Prize" caption above the dollar amount so a cold visitor reads it as competition prize money, not decoration.

- The leaderboard ranking explainer speaks to cold visitors: it names Bortz Age and Pheno Age as blood-test aging clocks in plain words and always ends with a "How ranking works" link to the ruleset. Every leaderboard view variant keeps that link — it is the main docs-discoverability path from the product surface.
- The scoreboard is the product surface, so its table header and explainer must make the scoring method inspectable without a docs detour: state the active score, clarify that lower/more negative age deltas rank higher, expose proof/result counts in rows, and keep tie-breakers/ruleset one click away. On phones, never let the score column clip offscreen; rank, athlete face/name, proof/result evidence, and score must all remain visible together.
- Standalone desktop leaderboard pages should land as standings, not title posters. Preserve athlete faces and evidence marks, but keep title spacing and row density tight enough that several real competitors are visible above the fold on common laptop heights.
- Desktop standings use compact page chrome. The leaderboard is the product, so `/leaderboard` should use a small wordmark header rather than the tall public masthead; preserve the dark sport identity and a visible Apply CTA, but spend vertical space on filters, search, rows, and scores.
- Wide desktop leaderboard pages should use a wider official-record lane than prose pages. The standings table is the product surface; on 1440px+ screens it should feel like a full competition board, not a narrow document column floating in paper.
- Tablet leaderboard landings should not inherit phone-only title padding. At 641-900px, keep the page title compact enough that search, filters, and the first rows enter the first viewport; reserve the extra title headroom only for phones where the floating CTA shares the row.

- Trust cues should be concrete and close to the competition surface. Desktop first viewport, podium cards, leaderboard rows, and athlete modals should expose real athlete faces, public-record/profile actions, proof counts or proof links, biomarker/result counts, ruleset/source links, prize-pool cues, and labeled press/contact affordances. Avoid empty future-facing trust placeholders such as "sponsor coming soon" in primary tables.

- Dates rendered inside English UI copy are pinned to an English locale (`en-US`), never the browser locale; a Hungarian or German visitor must not see "jún. 9." inside an English sentence.

- Focus rings are part of the green system everywhere (green border plus `rgba(67, 238, 131, ~0.25)` ring). Amber/gold stays reserved for crowd/Guess-My-Age features and Amateur marking, never for interaction states.

- Full-bleed entry posters (play hub) reuse the hero's scoreboard language: dark lit surface with the faint green sheen, rotated phoenix watermark, display-font uppercase headline with one green accent word, and a small letterspaced green kicker. Never ship a flat photo/JPG poster where a brand-language CSS poster can carry real text.
- Desktop entry screens must use a side-by-side decision composition: brand poster plus athlete-access actions visible together and top-aligned as one decision surface. Short landscape and small-desktop entry screens must not let poster art push the primary action below the first viewport, and tall desktops must not collapse into a lonely centered poster with large dead space.
- Mobile entry screens must expose both athlete-access paths before the fold on 320px-class phones: `Apply to compete` and `Manage athlete record` are the product decision, while proof/rules links and footer navigation follow after them. Poster art may set the sport tone, but it must not consume the first viewport at the cost of the two real actions.
- Application track cards must show the action before long proof or biomarker requirement lists on tablet and desktop. A visitor choosing between Pro and Amateur should see track identity, ranking context, entry requirement, and CTA before supporting details.
- The desktop application track-choice page uses compact masthead chrome, not the full public homepage masthead. The brand must remain unmistakable, but the first viewport should spend most of its height on the Pro/Amateur decision, CTAs, and proof requirements.

- Overlay layering scale: page chrome (header band, mobile drawer, sticky PLAY) lives at z-index 1000–1002; the athlete details modal owns the whole viewport above the chrome at 9000; top-level overlays that can stack on the modal (proof viewer, guess-my-age) live at 10000+. New overlays must pick a slot in this scale instead of inventing local z-index values.

- Page titles need clear air under the dark chrome: shared `h1` rules give titles a top margin and `scroll-margin-top` so fixed bars never cover them. Pages that auto-scroll on load (leaderboard, onboarding steps) must land with breathing room below the sticky bar — leaderboard matches the title's margin and scroll-margin so the scroll stops exactly at the header band's bottom edge; the onboarding scroll subtracts a fixed offset in `main-progress-bar.html`.

- The play application pages (`character-selection`, `character-customization`, `proof-upload`, `edit-profile`) use the tall scrolling header, not a fixed bar. First load should land with the compact header fully visible and the task title below it; do not auto-scroll these pages into a half-clipped brand band.

- The footer mirrors the header band's treatment (faint green sheen plus vertical gradient) so the dark chrome bookends the page consistently.

- Direct-tap controls in mobile, toolbar, footer, and modal contexts should generally be easy to hit, with about a 44px target when space allows.

- Controls that belong to the same flow should share the same visual language: inherited font, consistent height, modest radius, light border, and a visible focus state.
- Button and control hover states should not enlarge the control. Use color, a 1px lift, a tighter shadow, underline, or ring feedback. Scale is reserved for media reveal moments where the enlarged object has a fixed frame and cannot disturb nearby data.
- Never define broad element-level hover styles such as `button:hover` for branded behavior. Hover, focus, active, disabled, and loading states must be scoped to the component class so one page cannot accidentally recolor or resize unrelated controls.

- Component state contract: buttons, tabs, filters, inputs, and autocomplete rows need default, hover, focus-visible, active/selected, and disabled states. Primary actions use the green gradient pill; secondary actions use sage/outline surfaces; destructive controls use `--danger-*`; disabled controls use the gray-beige disabled tokens and no depth.

- Autocomplete/listbox panels are official-record surfaces: warm card background, full hairline border, `--surface-shadow-lifted`, green scrollbar thumb, 44px rows, green-tinted hover/active row, and brand-green highlighted match text. Do not ship cold white/slate dropdowns or heavy floating shadows.

- Image, crop, proof, and biomarker preview frames use `--surface-border`, `--radius-card`, and `--surface-shadow` or `--surface-shadow-lifted`; avoid the ghost-card combination of a full border plus large 16px+ blur unless the surface is a modal or dark hero poster.

- Helper, confirmation, validation, and empty-state copy should be grouped in compact light panels when grouping improves scanning or anchors the message to nearby controls.

- File, proof, and profile-image previews should stay inside bounded frames with `object-fit: contain` when unusual aspect ratios are likely.

- Autocomplete and suggestion menus should appear as padded floating panels with clear row hover/focus states; on constrained viewports, avoid covering the next likely action.

- Dense table and list hovers should not change font size or reflow neighboring content. Prefer color, underline, surface, edge, shadow, or slight lift for feedback.

- Empty states inside data tables should be compact and include an obvious recovery action whose shape matches nearby controls.

- Filter and segmented controls should make active, clearable, and unavailable states visually distinct without relying on a tiny badge alone.

- Disabled pills read as "switched off", not broken: a gray-beige fill clearly darker than the paper surface (`#e2e1d6`), muted gray-green text (`#97a098`), a quiet hairline border, and no depth cues. Disabled text is intentionally low-contrast; never leave a disabled control on the same fill as an enabled secondary control.

- The bioage calculators' result numeral is a brand moment: it uses the display font (single weight, no negative letter-spacing) with the brand green gradient, like the hero score and modal numerals. Its performance color variants stay in the brand family (greens and the amber/gold secondary), not Material/Flat-UI palettes.
- Bioage form fieldsets use full-width official section labels with a quiet divider, not floating pill legends. Keep the semantic `legend`, but avoid the native fieldset border cut-through by using an inset record-frame and a full-width legend header. Legends should organize required data groups without adding extra rounded chips or competing with the form controls.

- Mobile foreground surfaces such as drawers and full-screen viewers should have clear close targets and enough backdrop contrast that the active surface is unambiguous.

- Long user-generated labels, names, and button text should wrap without clipping; icons in action buttons should stay in fixed slots when labels wrap.

- Badges are compact visual tokens. Keep badge groups predictable and avoid hover behavior that turns them into unstable or overly tall content.
- Badge hovers use a tiny lift/ring/shadow, not size growth or pulsing. Utility badges such as personal links stay in the green/ink brand family; cyan/pink badge colors require a specific semantic reason and should not be used for generic links.
- Athlete badges are record chips, not game achievements. Keep shared badge shapes simple, flat, and token-based; avoid shiny inset highlights, novelty hex/circle shape mixing, saturated arcade fills, and heavy mini-shadows. Podium metals and true semantic statuses may stay distinct, but ordinary badge families should support rank/score rather than compete with them.
- Leaderboard table badges are evidence marks, not the content hierarchy. Keep them quiet, compact, and lower-contrast than athlete name, face, rank, and score; save larger labeled badge treatments for the athlete modal where the user is inspecting the record.
- Dense leaderboard row actions such as press/contact chips must not set vertical margins that inflate every table row. In standings tables, row action affordances sit inside the cell padding and stay secondary to rank, face, athlete name, and score.

- Mobile modal content should prefer readable stacked sections over preserving desktop columns; avoid nested scrolling when stacked content can use the modal's main scroll.

- Mobile athlete modal spacing is compact by default: rank/score/proof summary, public-record links, and badges should fit as a tight record header before the bio and long evidence sections. Badge grids on phones use compact tokens and two-line labels so credentials support the record without pushing proof/details too far down.

- Standalone pages without the shared partials (privacy policy, error pages) still follow the surface law: content lives on the light paper surface with ink text and green accents; dark belongs to chrome and posters only. Their local CSS copies must stay in tune with the site tokens.

- Leaderboard sidebar filter glyphs are monochrome icons that inherit the label's color through hover/active states — never fixed-color emoji inside the disciplined filter chrome. Country-flag emoji are the exception (flags are their own identity), and form `<option>` text keeps emoji because options cannot render HTML.
- Expanded leaderboard filters are official league-control surfaces, not generic drawers: use the dark chrome title bar with the trophy/Leagues identity, compact warm-paper filter rows, and an integrated Done/close control. Inactive hidden controls such as clear-filters must not reserve blank space.
- Mobile leaderboard drawer headers must fit at 320px with the title and Done/close control on one line. Do not allow active-state indicators or close buttons to force labels such as `Leagues` to wrap or clip.
- The standalone standings board should not show a collapsed filter rail when the toolbar already has a filter button. Default leaderboard state is the table-first product view; the Leagues panel appears only after the user explicitly opens filters, then closes back to a clean table.
- Opening leaderboard filters must preserve the current official-standings composition. Do not auto-scroll the toolbar to the top on desktop if it causes the compact header to cover or slice the page title.
- Selected leaderboard filters represent current league scope, so their active state uses the green/current-state system. Reserve amber for Amateur, crowd, prize, or new-entry semantics; do not use amber as the generic selected-filter color.

- The generated docs pages' mobile contents-nav remains an official index panel, not a loose tag cloud: keep the "Official index" label visible, use compact bordered link chips that wrap long entries inside the panel (`white-space: normal`, `overflow-wrap: anywhere`), and never let timeline-length labels run off the viewport edge. These styles live in the `MarkdownPageGenerator` template, not in the generated files.

- Every interactive control must have an accessible name. Icon-only links and buttons need `aria-label` or visible text; decorative icons inside them should be `aria-hidden="true"`. Form inputs, selects, textareas, file inputs, and visually hidden upload controls need a real `<label>`, `aria-label`, or `aria-labelledby`. Do not ship focusable decorative duplicate links.

- Dialogs and drawers must honor their basic contract before visual polish: visible close target, Escape-to-close except while intentionally blocking a sub-flow, focusable controls with names, and a clear backdrop/active surface relationship. Athlete modal checks must account for the first-time Guess My Age overlay and then verify the underlying record modal.

- Scroll anchors are part of navigation. Only link to hash targets that exist in the rendered page, and keep linked sections reachable below fixed chrome with `scroll-margin-top` or equivalent spacing.

- Final browser safety checks should keep using the repo-owned .NET/Playwright path and write artifacts under `.artifacts/`. A clean final lock pass means no console/page errors, no failed local requests, no broken internal links, no unnamed controls, working modal/drawer flows, working scroll anchors, and desktop/mobile screenshots for the major changed surfaces.

- Sparse official rankings need intentional framing. If a leaderboard view has only a few qualifying athletes, show a compact eligibility note with the qualifying count and reason before the table; do not let a short field read as a broken or unfinished load.
- Empty public live boards should frame the eligibility condition before the data exists. Do not lead with `0 people`, `0 check-ins`, or candy-colored score bands; use pending/live status language, restrained score tokens, and a compact official-board empty state until qualifying rows appear. On wide screens, keep the empty-state card to a readable measure instead of stretching one message across the whole board.
- Empty board states are still board states, not cards inside cards. When a public board has no rows yet, use the board shell itself for the status message, eligibility condition, and next live cue; avoid a centered placeholder panel floating in a large empty table area.
- Competition empty boards should still expose the rules that make the board real. For challenge boards, show compact facts such as eligibility, ranking basis, and tie-breaker near the empty status instead of leaving a large blank body.
- Dense day-grid boards may tighten columns at medium landscape widths, but they must never clip silently. At phone widths, convert rows into labeled cards; at tablet/small desktop widths, keep participant, score context, and all day cells readable within the visible board or an explicitly scrollable frame.

- Mobile returning-user flows are task-first. Put the primary input and continuation action before large illustrative media; keep the illustration as brand support after the task controls, not as the first obstacle.
- Returning-athlete lookup should feel like record access, not character selection. Do not show generic placeholder athlete imagery in the default state; reserve athlete photos for a selected real athlete.
- Returning-athlete lookup starts with the search task. Do not place a row of explanatory tool chips above the athlete field; proof, results, and profile tools are supporting context below the lookup until a real athlete is selected.
- Returning-athlete lookup should not autofocus the search field on first paint. Let the user choose the input deliberately so the page does not open with a stale focus ring or mobile keyboard before the record-access context is read.
- Calculator and onboarding pages should not force focus into a field on first paint. Move focus only after an explicit step change, validation error, or user action; otherwise mobile screens should open as a composed page, not an active form control.
- Desktop returning-athlete lookup uses a two-column official-access composition: record context and permitted tools on the left, lookup/search/action panel on the right. Do not collapse it into a narrow centered mobile stack on desktop.
- Returning-athlete dashboards are action surfaces, not profile galleries. On desktop, place the athlete identity/photo and record actions side by side so all actions are visible in the first viewport. On 320px-class phones, keep the athlete image compact enough that the primary record actions are visible before the fold.
- Returning-athlete edit forms are form-first. Keep the real athlete face for confidence, but size it as context on phones so division, country, public link, and media/contact fields begin in the first viewport.

- Mobile scrolled chrome gets one join CTA. When the compact sticky wordmark is active, the original in-header join button must not remain visible as a duplicate or contextless black-band CTA.
- Mobile public chrome should identify the sport without consuming the page. On docs, challenge boards, legal/error pages, and task-entry pages, the logo/wordmark/CTA header stays compact enough that the first content module is visible immediately on 320-390px phones; reserve poster-scale first viewports for the homepage hero and intentional entry posters.

- Footers and long secondary navigation need grouping labels when there are many links. Preserve the useful link set, but separate project/trust links from social/community links so the footer routes instead of becoming an icon dump.
- Contribution modules must feel like official competition infrastructure. Keep prize progress, QR code, and copy/address affordances in a tight transaction layout; the QR supports the prize pool and should not dominate the card or float as disconnected decoration.

- Full event archives should keep their dense sports-ticker rows, but long public feeds need date separators or equivalent wayfinding. The pulse header explains recent activity; the archive below must remain scannable by day without turning events into oversized cards.

- Mobile application steps are task-first. On first entry to `/apply`, required fields and the next/back controls come before large brand imagery or progress-art panels; phoenix/arena imagery supports the flow after the task controls instead of consuming the first viewport.
- Track-choice comparisons must match the decision hierarchy already shown in the CTAs. Pro is the competition-forward primary path and should appear before Amateur in visual order; Amateur remains available as the quieter secondary path.
- Application confirmation and review states are official process surfaces. Use received/review/next-step status language and restrained record panels; do not use memes, unrelated jokes, or waiting-room illustrations where the user expects competition administration.

- Athlete modals open as competition records, not profile galleries. The first viewport should show the athlete face/name, current rank, active score, and proof count before badges, bio, charts, events, or deep stats. Badges remain supporting credentials unless they directly explain the active rank.
- Athlete modal proof navigation must land on proof, not adjacent record noise. Keep public proof images before the event/highlights feed in the modal flow so the Proofs control opens directly to inspectable evidence.
- Athlete modal chart sections are evidence panels. Keep the clock switch flat and record-like, keep the radar/chart large enough to read on phones, and avoid oversized pill controls or soft capsule frames that make proof data feel decorative.
- Age-estimation and proof-inspection states are trust surfaces, not novelty games. Use immediate readable labels, compact official panels, green primary actions, and full-screen proof overlays above the modal stack; avoid typewriter/boom reveals, oversized playful headings, delayed controls, meme redirects, confetti/icon floods, or overlay effects that make inspection feel less credible.
- Age-estimation modals should size to the inspection task. Do not force a near-full-height dark panel when the portrait, slider, and actions need less space; empty modal chrome reads unfinished and weakens trust.
- Proof inspection should start with the document evidence near the top of the viewport below a real close button. Do not vertically center tall lab reports inside a large empty black field or shrink them to fit the whole page at once; allow the proof surface to scroll naturally at a readable document width.

- Mobile league-choice screens must expose all primary paths before detailed comparison cards. Pro remains the competition-forward green action, Amateur remains the quieter secondary path, and long biomarker detail belongs below the immediate choice.

- Nested progress inside a page must name the work, not just show dots. The global three-step journey can stay compact in chrome, but calculator sub-steps need visible labels such as `Age & date` and `Biomarkers` so users understand what the inner control changes.
- Task-page headings name the current work, not the step number. Put sequence in progress controls (`Application checklist`, `5 steps`, dots) and keep H1s as readable task labels such as `Athlete details`, `Proof upload`, and `Final details`.

- Error pages recover before they illustrate. On mobile, the title, short explanation, and primary recovery action must appear before large images; use `View leaderboard` as the primary 404 escape because the standings are the product center, with Home as the secondary route.

- Press and media-kit asset previews must show the actual asset without accidental cropping. Logos use contained square presentation frames, team/person images use bounded contain frames, and non-image files use compact file-type tokens; never let brand marks fill a card as a random crop.
- Legacy media-kit campaign art is archive material, not a live brand surface. Keep it downloadable, but show it as compact archive/file rows rather than large visual cards so old copy, buttons, or layouts cannot compete with current official marks and team assets.
